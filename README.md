# Second Brain 工程认知系统使用说明

这是一个面向 Java 游戏后端、JVM、Runtime System、分布式系统、性能诊断、故障分析和 AI Engineering 的 Obsidian Vault。

它不是资料收藏夹，而是长期工程认知基础设施。核心目标是把学习、源码阅读、项目实践、线上故障、性能压测、工程决策和 AI 对话沉淀为可复用的工程资产。

座右铭：独立思考，明辨是非。

## 1. 系统入口

优先从这些文件开始：

- `Home.md`：知识库首页，所有长期主线和入口都从这里进入。
- `00 - 系统说明/Vault 初始化总览.md`：完整设计总览，按 15 个部分说明整个系统。
- `00 - 系统说明/最小可用版本 MVP.md`：前两周最小使用方式。
- `00 - 系统说明/知识流转流程.md`：从输入到工程资产的处理流程。
- `00 - 系统说明/AI 使用工作流.md`：AI 对话如何沉淀，如何避免垃圾堆。

## 4. 每日使用流程

1. 遇到任何输入，先放入 `00 - Inbox`。
2. 如果是工程现场问题，优先创建故障、性能、实验或 ADR 笔记。
3. 如果是学习材料，不直接收藏全文，只提炼自己的判断、问题和验证计划。
4. 如果是 AI 回答，只保存经过验证的部分。
5. 当某条知识能指导行动时，沉淀为 Checklist、Prompt、ADR 或 Runbook。

推荐每天只做轻量记录，不追求完整整理。

## 5. 每周使用流程

每周至少做一次清理和复盘：

1. 清理 `00 - Inbox`。
2. 删除无价值材料。
3. 把有价值内容拆成概念、机制、案例、ADR、实验或 Checklist。
4. 更新相关 MOC。
5. 新增至少 1 个工程资产。
6. 使用 `Templates/每周复盘模板.md` 写周复盘。

周复盘重点不是“做了什么”，而是：

- 本周解决了什么真实问题？
- 哪些知识变成了工程资产？
- 哪些判断被验证或证伪？
- 哪些 AI 输出可靠或不可靠？
- 下周要构建什么？

## 7. 笔记创建原则

每篇长期笔记都应该能回答：

- 它解决什么问题？
- 机制是什么？
- 边界在哪里？
- 失败场景是什么？
- 有什么 tradeoff？
- 如何验证？
- 能否沉淀为工程资产？

不要因为一个名词就创建空笔记。先放在当前笔记中，等它出现第二次，或需要独立推理时再拆出来。

## 11. 双链规则

双链用于表达具体关系，标签用于集合检索。

应该使用双链的场景：

- 一个机制支撑一个故障结论。
- 一个实验验证一个 ADR。
- 一个性能案例沉淀出一个 Checklist。
- 一个 AI Prompt 被某个项目实践验证。
- 一个概念会被多个 MOC 复用。

完整说明见：

- `00 - 系统说明/双链与归档规则.md`

## 12. AI 使用规则

AI 原始回答不要直接进入长期知识库。

只保存：

- 验证过的解释。
- 有证据支撑的判断。
- 可复用 Prompt。
- AI 犯错模式。
- 可转化为 Checklist、ADR、实验或案例的内容。

不要保存：

- 未验证的大段回答。
- 常识性解释。
- 没有上下文的 Prompt。
- 无来源、无证据、不可复现的结论。

完整说明见：

- `00 - 系统说明/AI 使用工作流.md`

## 13. 工程资产化标准

一条知识只有满足以下条件之一，才算工程资产：

- 能减少未来排障时间。
- 能指导架构决策。
- 能降低上线风险。
- 能复用到多个项目。
- 能作为 Checklist 执行。
- 能作为 Prompt 稳定产出结果。
- 能作为实验结论支撑判断。

工程资产优先放入 `09 - 工程资产`，并挂到对应 MOC。

## 15. 维护节奏

- 每日：轻量捕获。
- 每周：清理 Inbox，新增工程资产，写周复盘。
- 每月：整理 MOC，合并碎片，复盘能力增长。
- 每季度：调整技术主线，盘点资产，确定下一阶段重点。

判断这个 Vault 是否有效，不看笔记数量，而看：

- 你是否更快定位问题。
- 你是否更清楚架构取舍。
- 你是否减少重复犯错。
- 你是否能把经验复用到下一个项目。
- 你的 AI 协作是否更可靠。
