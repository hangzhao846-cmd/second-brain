# 总体设计思路

这个 Vault 的核心不是“存知识点”，而是积累长期工程判断力。目录只负责粗粒度归位，真正的知识组织依赖 MOC、双链、标签、复盘和工程资产化流程。

## 设计原则

1. 以能力为主线：Java/JVM、游戏后端、Runtime、分布式、性能诊断、故障分析、AI 工程。
2. 以问题为入口：每篇长期笔记都应回答“它解决什么工程问题”。
3. 以资产为出口：学习资料应逐步升级为 Checklist、ADR、模板、案例、Prompt、工具脚本或架构判断框架。
4. 每个主题都要追问机制、边界、代价、失败场景、tradeoff 和实践验证。
5. 少建目录，多用 MOC。目录用于生命周期和领域边界，MOC 用于认知结构。

## 顶层目录说明

```text
00 - Inbox/
01 - MOC 地图/
02 - Java 与 JVM/
03 - 游戏后端/
04 - Runtime 与系统基础/
05 - 并发 IO 与网络/
06 - 分布式系统/
07 - 性能与故障诊断/
08 - AI Engineering/
09 - 工程资产/
10 - 项目与实验/
11 - 复盘与决策/
12 - 输出与写作/
Templates/
99 - Archive/
```

| 目录 | 用途 | 放什么 | 长期能力关系 |
|---|---|---|---|
| `00 - Inbox` | 快速捕获 | 临时想法、资料摘录、AI 对话、排障现场记录 | 降低捕获成本，避免中断工作流 |
| `01 - MOC 地图` | 知识导航 | 主题 MOC、资产索引、问题地图 | 形成长期认知结构 |
| `02 - Java 与 JVM` | Java/JVM 深水区 | JMM、GC、JIT、类加载、诊断工具、源码阅读 | 强化 Java 后端底层判断 |
| `03 - 游戏后端` | 游戏服务器领域知识 | 网关、场景、战斗、AOI、状态同步、跨服、运营工具 | 积累 MMORPG 架构经验 |
| `04 - Runtime 与系统基础` | Runtime 机制 | OS、调度、内存、系统调用、Runtime 设计 | 理解 JVM/游戏服/AI Infra 的底层共性 |
| `05 - 并发 IO 与网络` | 高并发基础 | 线程模型、锁、队列、Netty、IO 模型、协议 | 支撑高并发和低延迟系统设计 |
| `06 - 分布式系统` | 分布式工程判断 | 一致性、复制、缓存、消息、限流、容错 | 提升架构权衡能力 |
| `07 - 性能与故障诊断` | 问题解决能力 | 故障复盘、性能案例、压测、监控、排查路径 | 把经验沉淀成可复用方法 |
| `08 - AI Engineering` | AI 工程实践 | Prompt、Agent、RAG、评估、上下文工程 | 建立 AI 辅助工程能力 |
| `09 - 工程资产` | 可复用产物 | Checklist、ADR 模板、架构模板、Prompt、工具脚本 | 把知识转化为工程生产力 |
| `10 - 项目与实验` | 实践现场 | 项目记录、实验记录、PoC、压测记录 | 用实践校验知识 |
| `11 - 复盘与决策` | 反思与选择 | 周/月/季度复盘、ADR、项目复盘、学习复盘 | 建立判断闭环 |
| `12 - 输出与写作` | 对外输出 | 文章草稿、分享提纲、技术总结 | 倒逼结构化表达 |
| `Templates` | 笔记模板 | 各类笔记模板 | 保证记录质量一致 |
| `99 - Archive` | 归档 | 过期资料、完成项目、废弃想法 | 保持主系统清爽 |
